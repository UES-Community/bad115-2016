Poseer índices especializados no garantiza un alto rendimiento si el **Optimizador de Consultas Basado en Costos (Query Optimizer)** decide ignorarlos. Este motor estima el costo de E/S (I/O) y los ciclos de CPU de múltiples planes de ejecución alternativos, eligiendo la ruta óptima para resolver la consulta.

---

### Estimación de Cardinalidad y Selectividad

El optimizador toma decisiones apoyándose en dos métricas estadísticas fundamentales:

1. **Cardinalidad:** Representa el número de valores únicos y distintos en una columna. Una columna de "Género" (M/F) tiene baja cardinalidad, mientras que un "DNI" o "RFC" posee una cardinalidad altísima.
2. **Selectividad:** Es la proporción de filas de la tabla que se espera que devuelva un filtro específico. Se calcula como:
   $$\text{Selectividad} = \frac{\text{Cardinalidad de la condición}}{\text{Total de filas}}$$
   * Una selectividad cercana a $1.0$ (o $100\%$) significa que el filtro devuelve casi toda la tabla.
   * Una selectividad cercana a $0.0$ (o $0\%$) indica que el filtro es muy discriminatorio y devuelve una cantidad mínima de filas.

```sql
-- Caso 1: Selectividad excelente (cercana a 0%)
SELECT * FROM usuarios WHERE dni = '12345678A'; -- El optimizador usará el índice de inmediato.

-- Caso 2: Selectividad deficiente (cercana a 85%)
SELECT * FROM ordenes WHERE entregado = TRUE; -- Si el 85% de las órdenes están entregadas, el optimizador ignorará el índice y preferirá un escaneo secuencial (Full Scan) para evitar el costo de accesos aleatorios (Random I/O).
```

<Callout type="warning">
**Estadísticas obsoletas:** Si las estadísticas del optimizador no están actualizadas (tarea gestionada por procesos como `ANALYZE` en PostgreSQL), se produce una *Deriva de Estimación (Row-Estimate Drift)*, lo que lleva al optimizador a tomar malas decisiones (ej. realizar escaneos completos innecesarios o un uso ineficiente de índices).
</Callout>

### Índices Compuestos y la Regla del Prefijo Izquierdo (*Leftmost Prefix Rule*)

Un índice compuesto es una estructura única estructurada jerárquicamente sobre múltiples columnas (de 2 a 16 campos).

```sql
-- Creación de un índice compuesto
CREATE INDEX idx_ubicacion_clientes 
ON clientes (Pais, Estado_Ciudad, Edad);
```

#### Mecánica de Almacenamiento Jerárquico
El motor ordena los registros siguiendo estrictamente el orden de las columnas de izquierda a derecha:
1. Primero ordena físicamente por `Pais`.
2. Dentro de cada `Pais`, los registros se ordenan por `Estado_Ciudad`.
3. Finalmente, si hay valores idénticos en ambos campos, se ordenan por `Edad`.

#### La Regla del Prefijo Izquierdo
El optimizador **solo puede utilizar** este índice si el filtro de la consulta incluye la columna inicial (el extremo izquierdo) de la definición del índice.

* **Consultas que SÍ usan el índice:**
  * `WHERE Pais = 'México'` (Usa el prefijo `Pais`)
  * `WHERE Pais = 'México' AND Estado_Ciudad = 'CDMX'` (Usa `Pais` y `Estado_Ciudad`)
  * `WHERE Pais = 'México' AND Estado_Ciudad = 'CDMX' AND Edad = 30` (Usa el índice completo)
* **Consultas que NO usan el índice (o lo usan ineficientemente):**
  * `WHERE Estado_Ciudad = 'CDMX'` (Falla catastróficamente; ignora el índice porque falta la columna guía `Pais`).
  * `WHERE Pais = 'México' AND Edad = 30` (Usa el índice parcialmente para filtrar `Pais`, pero requiere realizar un escaneo interno para `Edad` debido a la ausencia de `Estado_Ciudad`).

#### El Impacto de los Operadores de Rango
Cuando se aplica un filtro de rango (`>`, `<`, `BETWEEN`, `LIKE 'abc%'`), el orden jerárquico se rompe a partir de esa columna, y los campos a su derecha en el índice no podrán ser utilizados para la búsqueda directa.

<Callout type="info">
**Axioma de Diseño:** Al crear índices compuestos, coloca siempre a la izquierda las columnas que se consultarán mediante comparaciones de igualdad estricta (`=`, `IN`), y a la derecha las columnas que usarán operadores de rango (`>`, `<`, `BETWEEN`).
</Callout>

### Índices de Cobertura (*Covering Indexes*) y la Cláusula `INCLUDE`

Un **Índice de Cobertura (Covering Index)** es aquel que contiene todas las columnas requeridas por una consulta (tanto en el `SELECT` como en el `WHERE` y `JOIN`).

Si el índice cubre por completo la consulta, el optimizador aplica un **Escaneo Exclusivo de Índice (Index-Only Scan)**, evitando descender a la tabla base (Heap o tabla agrupada) para recuperar datos adicionales.

```sql
-- Si consultamos solo columnas indexadas:
SELECT Pais, Estado_Ciudad FROM clientes WHERE Pais = 'México';
-- Ejecuta un Index-Only Scan directo desde la memoria caché, con rendimiento estelar.
```

#### Optimización con la Cláusula `INCLUDE` (PostgreSQL / SQL Server)
Permite adjuntar columnas de datos directamente en los nodos hoja del árbol B+, sin utilizarlas para ordenar el índice ni participar en la estructura del árbol.

```sql
CREATE INDEX idx_clientes_busqueda 
ON clientes (email) 
INCLUDE (nombre, telefono);
```

* **Ventaja:** El árbol se mantiene ligero y con un alto *fanout* (ya que no se ordena por `nombre` o `telefono`), pero permite responder consultas que pidan estos datos mediante un *Index-Only Scan* sin recurrir a la tabla física.

### Diagnóstico de Consultas: `EXPLAIN` y `EXPLAIN ANALYZE`

La validación del diseño de índices no debe ser teórica; debe auditarse directamente en el motor:

* **`EXPLAIN`:** Muestra la ruta planificada por el optimizador y el costo estimado en base a estadísticas.
* **`EXPLAIN ANALYZE`:** Ejecuta la consulta en tiempo real, midiendo los milisegundos exactos consumidos en cada paso del plan y el número real de filas procesadas.

Esto permite identificar cuellos de botella y desajustes de cardinalidad entre las estimaciones estadísticas y el comportamiento real del motor de almacenamiento.
