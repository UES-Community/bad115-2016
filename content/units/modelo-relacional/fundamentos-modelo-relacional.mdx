El modelo relacional destaca por su uniformidad estructural y su desvinculación total respecto de las estructuras de almacenamiento subyacentes, lo cual permite que su esquema pueda ser concebido íntegramente mediante construcciones matemáticas formales. El universo de discurso en una base de datos relacional se compone fundamentalmente de dominios y relaciones, sobre los cuales se imponen reglas inherentes dictadas por la naturaleza de los conjuntos matemáticos.

### Dominios, Atributos y Normalización

* **Dominio:** En la base de la jerarquía conceptual se encuentra el dominio. Se define formalmente como un conjunto nominado, finito y homogéneo de valores atómicos, el cual encapsula el espectro completo de valores lícitos que una propiedad específica puede adoptar dentro del sistema. La existencia de un dominio es ontológicamente independiente de las relaciones.
  * **Declaración:** Pueden declararse por extensión (delimitando cada uno de sus elementos constituyentes) o por intensión (mediante un tipo de dato y restricciones algebraicas).
  * **Atomicidad:** El modelo relacional puro prohíbe terminantemente la inclusión de estructuras anidadas, matrices o arreglos dentro de un dominio único. Cada intersección de fila y columna debe albergar un único valor indivisible.
* **Atributo:** Cuando un dominio es implementado dentro de una relación específica, adopta el rol semántico de un atributo (la interpretación contextual del dominio dentro del esquema). Múltiples atributos pueden extraer sus valores de un mismo dominio (por ejemplo, el *límite de crédito* y el *saldo actual* provienen del dominio de números reales).

#### El Proceso de Normalización
La regla de atomicidad es el pilar del proceso de normalización:
* **Primera Forma Normal (1FN):** Garantiza que todos los atributos contengan únicamente valores atómicos (indivisibles), facilitando la aplicación de operadores algebraicos sin requerir recursividad.
* **Segunda Forma Normal (2FN):** Erradica las dependencias funcionales parciales, exigiendo que todo atributo dependa por completo de la clave primaria.
* **Tercera Forma Normal (3FN):** Elimina las dependencias transitivas entre atributos que no forman parte de ninguna clave.

### La Estructura de la Relación: Tuplas, Grado y Cardinalidad

Una relación, entendida desde su definición matemática estricta, se constituye como un subconjunto del producto cartesiano de una lista finita de dominios ![Notación algebraica](/content/modelo-relacional/image1.png). 

Físicamente se representa como una tabla bidimensional, conformada por:
* **Esquema o Cabecera:** El nombre identificador y el conjunto de pares atributo-dominio.
* **Cuerpo:** El conjunto de las tuplas instanciadas en un momento temporal.
* **Tupla:** La unidad mínima de información instanciada (ocurrencia, vector o registro) que modela un hecho único. Carecen de ordenamiento posicional implícito (tanto horizontal como vertical).

#### Conceptos de Grado y Cardinalidad
* **Grado:** Cuantifica el número estático de atributos (columnas) delineados en su esquema formal. Es una variable predecible durante el ciclo de vida del sistema, alterada únicamente mediante operaciones DDL (`ALTER TABLE`).
* **Cardinalidad (o Aridad):** Mide el número dinámico de tuplas (filas) presentes en su cuerpo. Experimenta fluctuaciones constantes impulsadas por transacciones DML (`INSERT`, `DELETE`), influyendo críticamente en el optimizador del motor.

**Ejemplo de una Relación (Tabla):**

| ID (Atributo 1) | Nombre (Atributo 2) | Saldo (Atributo 3) | $\rightarrow$ **Grado = 3** (3 Columnas estáticas) |
| :--- | :--- | :--- | :--- |
| 1001 | Ana Martínez | 2500.00 | $\leftarrow$ Tupla 1 |
| 1002 | Carlos Gómez | 120.50 | $\leftarrow$ Tupla 2 |
| **Cardinalidad = 2** | | | (2 Filas dinámicas en el cuerpo) |

### Marcos de Integridad Inmanentes y Restricciones Semánticas

El modelo relacional mantiene la consistencia de los datos mediante reglas axiomáticas:

1. **No Duplicidad de Tuplas:** En un conjunto matemático no pueden existir elementos idénticos. Para implementarlo se usa la **Clave Candidata (Candidate Key)**: un subconjunto mínimo de atributos capaz de identificar de forma unívoca a cada tupla. Debe cumplir con la *minimalidad* (si se extrae cualquier atributo, pierde su propiedad unívoca).
2. **Clave Primaria (Primary Key):** Es la clave candidata elegida por el arquitecto de datos. El resto se consideran claves alternativas.
3. **Regla de Integridad de Entidad:** Ningún atributo que participe en la clave primaria puede albergar valores nulos (`NULL`). De lo contrario, la entidad perdería su identidad y se violaría el axioma de no duplicidad.
4. **Clave Foránea (Foreign Key):** Conjunto de atributos en una relación referenciadora cuyos valores deben coincidir con la clave primaria/candidata de una relación referenciada. Sostiene la **Regla de Integridad Referencial**, evitando registros huérfanos.

#### Integridad de Transacciones (ACID)
La consistencia en la ejecución de transacciones se rige por las propiedades ACID:
* **Atomicidad:** La transacción se ejecuta completamente o no se ejecuta en absoluto (haciendo un `ROLLBACK`).
* **Consistencia:** Una transacción lleva a la base de datos de un estado válido a otro estado válido.
* **Aislamiento:** La ejecución concurrente de transacciones produce el mismo estado que si se ejecutaran secuencialmente.
* **Durabilidad:** Una vez confirmada una transacción, sus efectos persisten incluso ante fallas del sistema.
