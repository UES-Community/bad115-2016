El paradigma tradicional de asignación de privilegios se fundamenta en un bloqueo binario: un usuario tiene acceso completo a una tabla o carece de él por completo. Sin embargo, en aplicaciones modernas como el software multi-cliente (SaaS), se requiere restringir el acceso para que cada cliente vea únicamente sus propios registros de forma transparente.

Para solucionar este desafío de seguridad surge el **Control de Acceso de Grano Fino (FGAC - Fine-Grained Access Control)**.

---

### Seguridad a Nivel de Fila (Row-Level Security - RLS)

Integrada de forma nativa en motores como PostgreSQL y SQL Server, RLS permite aplicar políticas de seguridad a nivel de registros individuales. Al delegar esta validación a la base de datos, se evita implementar lógica de seguridad dispersa e insegura en la capa de la aplicación.

#### Mecánica de RLS:
1. **Habilitación:** Se activa en una tabla mediante:
   ```sql
   ALTER TABLE tabla_nombre ENABLE ROW LEVEL SECURITY;
   ```
2. **Definición de Políticas:** Se establecen reglas utilizando `CREATE POLICY`.
3. **Reescritura de Consultas:** Cuando un usuario realiza una consulta, el motor agrega dinámicamente predicados de filtrado (cláusulas `WHERE`) en segundo plano según la identidad del usuario (`current_user()`).

#### Cláusulas de las Políticas RLS:
* **USING:** Aplica para consultas de lectura y actualización (`SELECT`, `UPDATE`, `DELETE`). Si la condición es falsa, los registros simplemente se ocultan (como si no existieran), sin levantar errores.
* **WITH CHECK:** Valida las operaciones de inserción y modificación (`INSERT`, `UPDATE`). Si el nuevo registro viola la política, el DBMS bloquea la operación y lanza una excepción.

#### Modos de Combinación:
* **Permisiva:** Las políticas se suman mediante operadores `OR` (amplía el acceso).
* **Restrictiva:** Las políticas se aplican mediante operadores `AND` (añade barreras adicionales).

### Bases de Datos Virtuales Privadas (VPD - Oracle)

En el entorno de Oracle Database, FGAC se implementa de forma altamente flexible a través de la **Base de Datos Privada Virtual (VPD)**, administrada mediante el paquete de sistema `DBMS_RLS`.

#### Funcionamiento de VPD:
A diferencia del RLS estático, VPD asocia la tabla a **funciones generadoras de predicados** escritas en PL/SQL:
1. Cuando se consulta la tabla, el motor ejecuta primero la función PL/SQL.
2. Esta función analiza el contexto del usuario (IP, rol, región) y devuelve dinámicamente un predicado en forma de cadena de texto (ej. `'zona = SUR'`).
3. El DBMS añade este predicado a la cláusula `WHERE` del usuario.

* *Ejemplo:* Si un cajero del sur escribe `SELECT * FROM cuentas;`, VPD la reescribe internamente como:
  ```sql
  SELECT * FROM cuentas WHERE zona = 'SUR' AND clase = 'TESORERIA';
  ```

#### Ventajas de VPD:
1. **Centralización Hermética:** La lógica de seguridad reside exclusivamente en la base de datos, no en las APIs ni en el frontend.
2. **Omnipresencia y Evasión Cero:** Al operar en el compilador de SQL (*parser*), ningún acceso directo (vía ODBC o herramientas de administración) puede eludir la seguridad.
3. **Consolidación Estructural:** Permite almacenar información de empresas rivales en el mismo esquema lógico sin riesgo de fuga de datos, evitando crear particiones físicas o instancias independientes (*sharding*).
