Las políticas de integridad garantizan la consistencia interna de los datos, pero delegan la protección perimetral a la seguridad lógica. Esta se instrumenta mediante el **Lenguaje de Control de Datos (DCL - Data Control Language)**, que define qué usuarios tienen autorización para consultar o modificar el esquema relacional.

---

### Comandos Fundamentales de DCL

* **GRANT (Concesión):** Otorga privilegios operativos a un usuario o rol. Al usar `WITH GRANT OPTION`, se faculta al receptor para heredar y otorgar esos mismos permisos a otros usuarios.
* **REVOKE (Revocación):** Elimina permisos concedidos previamente. Si se añade la palabra clave `CASCADE`, se revocan recursivamente los permisos otorgados por el usuario afectado a terceros.
* **DENY (Denegación):** Impone una prohibición explícita sobre un recurso (disponible en motores como SQL Server). **El DENY tiene precedencia absoluta sobre el GRANT**; si un usuario tiene un GRANT heredado por un rol pero un DENY individual, el motor bloqueará la operación.

---

### Categorías de Privilegios

La seguridad lógica se gestiona en dos niveles:

1. **Privilegios de Sistema:** Permisos globales para modificar la configuración del motor o crear recursos (ej. `CREATE DATABASE`, `CREATE USER`, `ALTER SYSTEM`, y roles administrativos como `SYSDBA`).
2. **Privilegios de Objeto:** Permisos enfocados en un elemento específico del esquema (ej. `SELECT`, `INSERT`, `UPDATE` o `DELETE` sobre una tabla, o `EXECUTE` sobre un procedimiento almacenado).

---

### Control de Acceso Basado en Roles (RBAC)

Para evitar la ineficiencia de asignar permisos de forma individual, los DBMS modernos utilizan **Roles**:
* Un **Rol** es un contenedor virtual al que se le asocian múltiples privilegios.
* Los usuarios reales se asignan como miembros del rol, heredando automáticamente todo el abanico de permisos configurados.
* Esto simplifica la administración, centraliza el control y reduce errores humanos en la asignación de accesos.
