Una vez estructurados y validados, los documentos XML requieren infraestructuras transaccionales para ser leídos, procesados y modificados por la lógica de negocio de las aplicaciones. La elección del modelo de análisis (*parsing*) impacta exponencialmente en el consumo de memoria, el rendimiento de la CPU y la mantenibilidad del código.

---

### Modelos de Análisis XML (*Parsing*)

Existen tres arquitecturas principales para procesar y parsear documentos XML en memoria de trabajo:

#### 1. Document Object Model (DOM)
Representa todo el documento XML como una estructura jerárquica de árbol cargada completamente en la memoria RAM.

* **Ventajas:** Permite navegación bidireccional libre, manipulación dinámica (insertar, editar, eliminar nodos) y ejecución nativa de consultas XPath sobre el árbol de objetos (`Node`, `Element`, `Text`, `Attr`).
* **Desventajas:** Presenta una complejidad de espacio prohibitiva de:
  $$O(N)$$
  El árbol en memoria puede consumir de 5 a 10 veces el tamaño físico del archivo XML original.
* **Caso de Uso:** Recomendado exclusivamente para la modificación de archivos pequeños o cuando se requiere conocer toda la jerarquía del documento simultáneamente.

#### 2. Simple API for XML (SAX)
Modelo basado en eventos guiado por empuje (*push model*). El analizador lee el archivo secuencialmente como un flujo de texto sin cargarlo en memoria.

* **Mecánica:** A medida que detecta etiquetas de inicio, texto o etiquetas de cierre, "empuja" notificaciones asíncronas llamando a funciones de retrollamada (*callbacks*) implementadas por el desarrollador (ej. `startElement()`, `endElement()`, `characters()`).
* **Ventajas:** Complejidad de memoria constante y mínima:
  $$O(1)$$
  Es idóneo para ingestar archivos gigantescos de varios gigabytes.
* **Desventajas:** Es estrictamente de solo lectura y unidireccional. La complejidad lógica se traslada al desarrollador, quien debe construir máquinas de estado manuales para no perder el contexto del nodo que está leyendo.

#### 3. Streaming API for XML (StAX)
Modelo basado en eventos guiado por extracción (*pull model*). En lugar de que el analizador dispare eventos ciegamente, es la aplicación la que toma el control e itera pidiendo el siguiente evento léxico.

```java
// Ejemplo conceptual del bucle de extracción StAX (Java)
XMLInputFactory factory = XMLInputFactory.newInstance();
XMLStreamReader reader = factory.createXMLStreamReader(new FileInputStream("datos.xml"));

while (reader.hasNext()) {
    int event = reader.next();
    if (event == XMLStreamConstants.START_ELEMENT) {
        if ("sensor".equals(reader.getLocalName())) {
            System.out.println("ID: " + reader.getAttributeValue(null, "id"));
        }
    }
}
```

StAX provee dos variantes de API para la lectura:
* **Cursor API (`XMLStreamReader`):** Funciona como un cursor deslizante de bajo nivel sobre el flujo de datos. Minimiza la creación de objetos en el Heap, logrando el máximo rendimiento.
* **Iterator API (`XMLEventReader`):** Envuelve los eventos en objetos fuertemente tipados (`XMLEvent`). Es de mayor nivel, soporta conductos (*pipelines*) y permite usar `.peek()` para inspeccionar el siguiente evento sin consumirlo.

### Vinculación de Datos (*XML Binding*) y ElementTree

#### JAXB (Java Architecture for XML Binding)
Automatiza la traducción entre documentos XML y objetos nativos de programación mediante anotaciones. 
* **`Unmarshalling`:** Transforma un XML de entrada en una instancia de clase Java.
* **`Marshalling`:** Convierte objetos de negocio de vuelta a formato de texto XML.
* **Desventajas:** Alta sobrecarga en CPU y rigidez si el XML cambia constantemente de esquema.

#### ElementTree (Python)
Combina ambas metodologías de análisis:
* **`parse()`:** Carga todo el documento como un árbol en memoria (estilo DOM).
* **`iterparse()`:** Actúa como un analizador incremental (*pull parser*), permitiendo liberar memoria dinámicamente conforme se procesan los eventos de cierre de nodos, conteniendo el uso de memoria en análisis de macrodatos.

```python
# Liberación de memoria con iterparse en Python
import xml.etree.ElementTree as ET

for event, elem in ET.iterparse("sensores_masivo.xml", events=("end",)):
    if elem.tag == "registro":
        procesar_datos(elem)
        elem.clear() # Libera el nodo de la memoria RAM
```

### Comparativa General de APIs de Procesamiento

| Dimensión Operativa | DOM (Document Object Model) | SAX (Simple API for XML) | StAX (Streaming API for XML) | JAXB (XML Binding) |
| :--- | :--- | :--- | :--- | :--- |
| **Modelo Arquitectónico** | Carga del árbol en memoria. | Flujo por eventos (*Push*). | Flujo por eventos (*Pull*). | Mapeo Objeto-XML. |
| **Dirección de E/S** | Lectura y Escritura. | Solo Lectura. | Lectura y Escritura. | Lectura y Escritura. |
| **Gestión de Estado** | Nativa y automática. | Manual (Compleja). | Procedimental (Sencilla). | Abstraída en clases. |
| **Uso de Memoria** | Alto ($O(N)$). | Mínimo ($O(1)$). | Mínimo ($O(1)$). | Medio-Alto. |
| **Soporte XPath** | Nativo y completo. | No soportado. | No soportado. | No aplicable directamente. |

### Arquitecturas de Persistencia XML

El almacenamiento persistente de XML se divide según la prioridad del modelo de datos:

#### 1. Bases de Datos Habilitadas para XML (*XML-Enabled*)
SGBD relacionales tradicionales (PostgreSQL, Oracle, SQL Server) adaptados para procesar XML.
* **Mapeo Relacional:** Descomponen el documento XML e insertan los valores en tablas y filas relacionales normales.
* **Almacenamiento Nativo XMLType / CLOB:** Almacenan el documento íntegro en columnas especiales de tipo texto o binario, indexando campos clave mediante expresiones XPath.
* **Caso de Uso:** Indicado cuando el 90% de los datos de la aplicación son estrictamente relacionales.

#### 2. Bases de Datos XML Nativas (*NXD - Native XML Databases*)
Sistemas de bases de datos diseñados específicamente para almacenar documentos XML como su unidad lógica de información (ej. eXist, Tamino, XIndice).
* **Mecánica:** Preservan la jerarquía original del documento, espacios de nombres y el orden de los nodos sin fragmentarlos.
* **Estructura:** Organizan la información en colecciones (análogas a carpetas o tablas relacionales) y optimizan la indexación de nodos para consultas dinámicas y masivas con **XQuery**.

## Ideas clave

<Callout type="info">
**DOM vs Streaming:**
Usa DOM cuando necesites consultar repetidamente el árbol en memoria o mutar su estructura. Prefiere SAX o StAX para la lectura secuencial de documentos masivos que no caben en la RAM.
</Callout>
