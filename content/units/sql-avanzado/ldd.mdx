El **Lenguaje de Definición de Datos (DDL - Data Definition Language)** provee la infraestructura para definir, alterar y eliminar los esquemas de la base de datos. 

A diferencia del DML, que modifica los registros del usuario, el DDL altera el **Diccionario de Datos (o Catálogo del Sistema)**. Este diccionario es un repositorio de metadatos protegido por el motor que almacena las estructuras de las tablas, tipos de datos, índices y restricciones de integridad.

---

### Comandos Esenciales de DDL

* **CREATE (TABLE / VIEW / PROCEDURE):** Edifica nuevas estructuras desde cero, definiendo de forma rígida los dominios de datos y las reglas de integridad de entidad (`PRIMARY KEY`, `FOREIGN KEY`).
* **ALTER (TABLE):** Realiza modificaciones sobre un esquema existente (añadir o eliminar columnas, expandir tipos de datos o inyectar nuevas restricciones) sin necesidad de reconstruir físicamente la tabla.
* **DROP (TABLE):** Elimina permanentemente la definición del esquema en el catálogo y destruye de forma irrecuperable los datos asociados en disco.

---

### Desafíos en la Migración de Esquemas DDL

Al migrar esquemas de bases de datos entre diferentes motores (ej. de Oracle a PostgreSQL o SQL Server), surgen asimetrías dialécticas importantes:

* **La Tabla DUAL (Oracle):** Oracle requiere proyectar desde una tabla virtual del sistema llamada `DUAL` al ejecutar funciones escalares sin origen físico (ej. `SELECT SYSDATE FROM DUAL;`). Al migrar, esta tabla debe ser eliminada o simulada en el motor destino si este no la requiere.
* **Tipos de Datos Temporales:** El tipo `DATE` en Oracle compila fecha y hora simultáneamente. En cambio, motores como SQL Server o Postgres separan de forma estricta los tipos `DATE`, `TIME` y `DATETIME`, requiriendo un mapeo cuidadoso.
* **Cadenas Vacías vs. NULL:** Oracle trata las cadenas de texto vacías (`''`) como valores `NULL`. Esto difiere del estándar ANSI y de la mayoría de motores, que evalúan las cadenas vacías y los nulos como estados lógicos independientes. Esta discrepancia puede corromper la lógica booleana de las aplicaciones al cambiar de base de datos.

## Practica guiada

<Callout title="DDL vs DML">
CREATE y ALTER definen la estructura; INSERT, UPDATE y DELETE modifican las instancias sin cambiar el esquema.
</Callout>
